# Тестовое задание на Unity + Mirror  
## Описание
Создать игру с p2p мультиплеером, где игроки могут подбирать предметы, использовать их и обмениваться ими.
Игрок может использовать кирку для добычи ресурсов, флягу с водой для восполнения энергии.
Также заложена система сохранений.

## Как открыть проект
Unity 6000.2.13f1  
Клонировать репозиторий и открыть проект.  
Все зависимости должны подругизться автоматически. Если что-то не загружается, самостоятельно установить:  
- Zenject (Extenject)
- Mirror
- Sprite 2D Package

Игра запускается со сцены Boot.  

## Игровой пайплайн и управление  
### Подключение
Выбираете адрес и порт, по умолчанию стоит localhost:7777.  
Хост нажимает "Хост" и выбирает файл сохранения, либо создаёт новый.  
Далее при запуске локации хост выбирает игрока, либо создаёт нового. 

Клиент вводит адрес и порт, и нажимает "Connect".  
Если успешно подключается, то выбирает игрока, либо создаёт нового.  

### Управление:  
- Esc: пауза
- Q: выбросить предмет
- Tab: открыть вкладку с инвентарём, где предметы можно перетаскивать между слотами с помощью drag'n'drop.
- ЛКМ: использование предмета
- E: подбор предмета
- WASD: передвижение  

## Код и архитектура
## Основные решения
Для максимальной крутости работы был выбран подход с авторитарным сервером + client prediction + reconciliation.
Инвентарь и прочие данные являются лишь данными и никак не синхронизируются.
Синхронизацией занимаются системы, которые имеют свою бизнес-логику и меняют данные через команды.  
Для DI используется Zenject.

Зачастую сетевые системы работают следующим образом:  
Клиент отправляет команду, что хочет совершить действие. У себя он сразу совершает действие предиктивно (client prediction).  
Сервер обрабатывает команду и валидирует её. В случае успеха отправляет всем клиентам подтверждение (confirm), либо отмену вызывающему клиенту (cancel).  
Если вызывающий команду клиент получил отмену, операция откатывается.  
Если клиенты получают подтверждение, они выполняют операцию, запускают анимации и т.д.  

Все важные действия игроков отправляются с гарантией порядка доставки.  

### Пример
Представим гипотетическую ситуацию. Два клиента одновременно нажали на подбор одного и того же предмета.  
Оба клиента предиктивно подобрали его и отправили команду на сервер.  
Сервер сначала получил команду от игрока 1, и отдал предмет ему. Далее он отправляет всем клиентам подтверждение,
что игрок 1 получил предмет.  
Далее сервер получил команду от игрока 2, но предмет уже занят, и поэтому он присылает отмену.  

Игрок 1 получает подтверждение своего действия, и ничего не делает, так как уже всё взял.  
Игрок 2 получает подтверждение действия, что игрок 1 взял предмет. 
Он отменяет свой действие, удаляет предмет из своего инвентаря и присваивает его игроку 1.  
Далее игрок 2 получает отмену своего действия. Своё действие он уже отменил, поэтому ничего не происходит.  

Таким образом гарантируется стабильность системы, а также отзывчивость на клиенте.  

## Папочки
Все игровые файлы лежат в Assets/Game.  
В папке Assets лежат ScriptableObject с данными.  
В папке Prefabs - префабы, Scripts - скрипты и т.д. соответственно.  

В папке Scipts:
- Animations: анимация для инструментов  
- Di: контекст Zenject с сервисами  
- GameSystems: тут большинство систем из п. Главные сервисы.  
- Network: тут EventBus, NetworkRigidbodyPredictable и скрипты для главного меню с подключением.  
- Player: тут управление игроком, инвентарь, синхронизация, выносливость, взаимодействие с предметами.
- - Controls: тут управление инвентарём, меню, передвижение и пр. А также PlayerInteractor.
- - Inventory: тут инвентарь. Здесь же предметы и события инвентаря.
- SceneManagement: здесь скрипт для синхронизации сцены при старте 
(для сохранения/загрузки важно, чтобы сначала вся сцена была синхронизирована, а затем создавался игрок) и StaticData. 
- Ui: здесь весь UI.

## Главные сервисы  
### IPlayerControls  
Инкапсулирует InputSystem и предоставляет интерфейс для подписки на события нажатий на кнопки.  

### INetworkObjectFactory  
*Scripts/GameSystems/Factories*  

Позволяет создавать объекты по сети, а также переопределяет хэндлеры спавна. Все заспавненные объекты
синхронизируются по сети и инжектят зависимости.  
Также предоставляет возможность добавлять событие, которое вызовется после спавна объекта. И событие, которое 
вызовется перед удалением объекта с гарантией, что объект будет заспавнен до того, как вызовется команда, или 
наоборот, ещё не будет уничтожен, пока команда не вызовется на сервере и всех клиентах.  
Например: перед уничтожением объекта выкинуть его из инвентаря.  

### IPlayerRepository  
*Scripts/GameSystems/Repositories*

Хранит список всех созданных игроков, их netId и компонентов. Также отдельно хранит ссылку на локального игрока.  

### INetworkServerPublishing  
*Scripts/Network/EventBus*

Просто класс с удобными методами для отправки сообщений с использованием NetworkServer.

### IPlayerActionCommandHandler
*Scripts/Player/Control*

Сервис для использования PlayerActionCommand. Если обернуть сообщение в PlayerActionCommand, то сервис 
гарантирует порядок доставки всех команд. Есть ограничение, что на одну команду один разработчик. 
Либо на одну команда для одного игрока один обработчик.  
Позволяет не создавать один хэндлер для всех команд игрока, а разбивать их по сервисам.

Например, DropItemCommand оборачивается в PlayerActionCommand и отправляется на сервер.  
На сервере сервис IPlayerActionCommandHandler разворачивает это сообщение и вызывает обработчик для игрока с указанным id.  
Если клиент быстро вызовет AddItemCommand и DropItemCommand, обернув их в PlayerActionCommand, 
то они гарантировано придут именно в этом порядке.  

### IEventBus  
*Scripts/Network/EventBus*

EventBus, которые публикует события локально в рамках одного клиента. Может быть использован 
для событий объектов, которых на сцене много, или которые спавнятся/удаляются. Например, для игроков и предметов.

### IMiningSystem  
*Scripts/GameSystems/Mining*

Система для добычи ресурсов. Внутри просто рейкастит, чтобы получит объект руды, и добавляет руду в инвентарь, если требуется.  

### IToolUsingSystem  
*Scripts/GameSystems/ToolUsingSystem*

Система для использования предметов. Поделён на два класса: ToolUsingSystem и NetworkToolUsingSystem.  
В одном находится бизнес-логика: валидация, проверки, изменение выносливости, вызов нужных методов инструментов.  
Во втором находится клиент-серверная логика с client prediction.  

### IGameStarter  
*Scripts/GameSystems/GameStarting*

Запускает сцену. Внутри содержит пайплайн:
- WorldSaver: загружает данные из сохранения, если запущен сервер и есть сохранения.  
- PlayerChooser: отображает интерфейс с выбором игроков.  
- PlayerSpawner: спавнит выбранного игрока или создаёт нового.  

### IGameFinisher  
*Scripts/GameSystems/GameFinishing*

Производит необходимые операции перед завершением игры. На данный момент это 
сохранение мира и отключение сервера с запуском главного меню.  

### IPlayerSaver и IWorldSaver  
*Scripts/SavingWorld*

Сохраняют или загружают данные об игроках или мире в целом.  
IPlayerSaver также сохраняет данные клиента, который отключился от сервера.  

### IMenusSystem  
*Scripts/Ui/MenusSystem*

Через эту систему запускается интерфейс. Одновременно может быть запущен тольок один интерфейс.  
Также включает/выключает отображение курсора.  

## Прочие важные компоненты  
### Inventory  
*Scripts/Player/Inventory*

Инвентарь игрока. Содержит данные и базовую бизнес-логику: прикрепить объект к руке, выкинуть и т.д.  
Не содержит логику для сетевого взаимодействия.  

### PlayerInteractor  
*Scripts/Player/Control/*

Взаимодействует с объектами (пока что только подбор предметов). Имеет клиент-серверную логику с 
client prediction и отменой. Вот он уже может синхронизировать инвентарь, путём отправки команд о том, 
что предмет был подобран.  

### NetworkRigidbodyPredictable  
*Scripts/Network/Physics*

Синхронизирует Rigidbody на клиентах. В отличие от стандартного NetworkRigidbody он делает это не постоянно, а
редко, и отправляет данные только при столкновениях, чтобы не нагружать трафик. 

### INetworkSerializable  
*Scripts/Network/PayloadTransfer*

Интерфейс, который даёт методы Serialize и Deserialize соответственно. 
Используется PlayerActionCommand'ом для сериализации данных и оборачивания в команду.

### NetworkInventorySync  
*Scripts/Player/Inventory*

При нажатии на Tab можно менять порядок предметов. Данный класс при применении нового порядка синхронизирует это 
с сервером.  

## По основным пунктам задания  
Инвентарь выступает лишь как данные, без логики синхронизации.  
При подбирании предметов работает PlayerInteractor, которые имеет клиент-серверную логику с client prediction и 
синхронизирует подбор предметов.  
При выбрасывании предметов работает NetworkInventoryController, который синхронизирует выбрасывание предметов.  
Он же синхронизирует изменение "выбранного активного" предмета игрока.  
Клиент отправляет данные о том, что он поменял выбранный предмет с тайм-аутом, чтобы активный скролл мыши не 
убивал трафик.  
При использовании предмета используется IToolUsingSystem в котором есть необходимая клиент-серверная логика с 
client prediction и отменой.  

Каждый предмет наследуется от BaseItem. Внутри у него базовая логика, плюс всякие методы на переопределение: 
OnInventoryAdded, OnInventoryRemoved, OnChoose, OnNotChoose.  
Есть интерфейс ITool - для предметов, которые можно использовать. Он используется в IToolUsingSystem. 
И абстрактный класс BaseTool, который его реализует. Там уже забита нужная логика и данные.  
Есть абстрактный класс BaseMiningTool - это BaseTool, который при использовании вызывает IMiningSystem, 
чтобы добывать руду.  
SimpleMiningTool и StaminaIncreasingTool - это кирка и вода соответственно.  
Все данные предметов и игрока выделены в ScriptableObject и вытащены в папку Assets/Game/Assets.  

При выбрасывании предмета все его данные с использованием INetworkSerializable синхронизируются.  
Предметы находятся в Assets/Game/Scripts/Player/Inventory/Items

## Пункты "Будет плюсом"  
### Анимации  
Сделаны простые анимации использования предметов. BaseTool автоматически включает нужные анимации при использовании.  

### Адаптивный UI  
UI слотов инвентаря, UI инвентаря на Tab, UI выбора игрока и сохранений, каждый из них создаётся динамически 
на основе тех или иных данных. Там используется ScrollView + LayoutGroup. Во время создания объектов 
используется IEnumerator и ожидание кадра, потому что большое количество вызовов Instantiate и изменение канваса 
может привести к вылету. После инициализации UI обычно отключает LayoutGroup, чтобы он не нагружал систему.  

### Расширяемость  
Сами предметы имеют различные методы на реагирование. Также есть IEventBus и события сервисов. 
События при изменении инвентаря, добавлении предмета или выбрасывании предмета и прочее. На каждое можно подписаться.

Для примера с грязной водой: два способа реализации.
1. Подписаться на событие выбрасывания предмета из инвентаря на стороне сервера, и там поменять воду на грязную.
2. Наследовать класс от воды для грязной воды, и в методе OnInventoryRemoved прописать логику загрязнения.  

### Сохранение и загрузка мира  
IWorldSaver сохраняет мир. Он использует BinaryWriter для чтения данных сразу из файла.  
Любой объект на сцене, которые реализует нужный интерфейс, будет сохранён и затем загружен.  
Игроки сохраняются отдельно в IPlayersSaver.  

Вообще игроки сохраняются отдельно и затем при загрузке мира любой клиент может выбрать, за какого 
сохранённого игрока он будет играть. Либо создать нового.  
Когда клиент отключается от сервера, сервер сначала сохраняет все данные клиента, а затем удаляет его.  
Сохранение мира на сервере происходит при нажатии кнопки Save & Quit в меню. Как в майнкрафт.  

Сохранение и загрузка данных использует persistentData и работу с Directory, поэтому антивирусник может ругнуться.  
Данные записываются в бинарном виде в формате .dat.

Сохранение и загрузка особо не протестированы, поэтому могут вызывать странные баги на стороне клиента.

## Проблемы  
Безусловно, есть спорные моменты в реализации. В том числе с синхронизацией данных, которые могут 
плохо работать при загрузке данных из файла сохранения, что связано с неправильным вызовом событий.  
В спешке некоторые сервисы несколько раз переписывались, поэтому могут быть ошибки.  
Также передача данных использует указание uint playerId в rpc методе, что может быть уязвимостью для хакера.  
Также я б заменил корутины на UniTask.  

В целом проблемы решаемы, но в рамках тестового задания и сроков оставлены в репозитории.
